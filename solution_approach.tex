\section*{Project Overview}
\addcontentsline{toc}{section}{Project Overview}

\textbf{MuleTrace (MTX)} is a high-performance Anti-Money Laundering (AML) classification and entity linkage platform built to detect money-mule accounts in real-time. The application combines advanced machine learning algorithms with rule-based compliance scorecards, explainable AI (XAI) narratives, and dynamic graph visualization. 

MTX resolves the classical banking trade-off between ML classification performance and regulatory transparency by outputting localized feature attributions alongside a dynamic Natural Language Generation (NLG) engine that generates human-readable forensic arguments.

\subsection*{Project Objectives}
\begin{itemize}[itemsep=3pt]
    \item Implement a robust ML pipeline addressing severe target class imbalance ($0.89\%$ positive mule rate).
    \item Audit raw transactional datasets to identify and neutralize temporal and index-based data leakages.
    \item Compress high-dimensional datasets containing 3,921 features into an optimized 168-dimensional hybrid feature set to meet the strict real-time banking ingestion latency of sub-2ms.
    \item Build a bimodal decision engine comprising precision-focused and recall-focused classifiers.
    \item Engineer a compliance sandbox using profile cloning to evaluate manual policy interventions (surcharges and mitigation discounts) without database contamination.
    \item Deploy a force-directed entity resolution graph linkage map using vis.js to isolate mule syndicate rings.
\end{itemize}

\subsection*{Technology Stack}
\begin{itemize}[itemsep=3pt]
    \item \textbf{Backend Ingest \& API:} FastAPI, Uvicorn (async server)
    \item \textbf{Machine Learning Engines:} XGBoost (extreme gradient boosting), LightGBM (light gradient boosting)
    \item \textbf{Data Processing & Auditing:} Pandas, NumPy, Scikit-learn
    \item \textbf{Frontend GUI & Visualizations:} HTML5, Tailwind CSS, vis.js (force-directed canvas)
    \item \textbf{Cloud Deployment:} Render Cloud Platform with dynamic port binding
\end{itemize}

\subsection*{System Processing Phases}
\begin{enumerate}[itemsep=2pt]
    \item Real-Time Streaming Data Ingestion \& Median Reconciliation
    \item Data Preprocessing \& Categorical Encoding Mapping
    \item Dual-Engine Model Inference (XGBoost vs. LightGBM)
    \item Sandbox State Check & Perturbation Scorecard Application
    \item Local Feature Attribution (LFA) & Explainable AI Weighting
    \item Natural Language Compliance Narrative Generation (NLG)
    \item Dynamic Graph Linkage Update & Automated SAR Template population
\end{enumerate}

\section{Compiler Preprocessing \& Data Leakage Resolution}

\subsection{Index/Sorting Leakage}
During initial model audits, we identified that the training dataset rows were sorted chronologically by the target variable (mule status). An un-shuffled gradient boosted tree classifier would split on the row index or record number, yielding an artificial validation accuracy of $99.9\%$ that drops to $50\%$ in production.
\\\textbf{Mitigation:} We dropped the index, shuffled the dataset using a random seed, and reset the index pointers before model training.

\subsection{Temporal Batch Leakage}
Features $F2230$ and $F3912$ acted as exact temporal separators (clean cases were clustered in specific months, while mule cases were clustered in others). The model learned to classify based on date batches rather than transactional behaviors.
\\\textbf{Mitigation:} We excluded $F2230$ and $F3912$ entirely from the training pipeline.

\subsection{Hybrid Feature Selection}
Running real-time inference on 3,921 features is computationally unfeasible under strict banking SLA limits (<2ms). We compressed the dimensions using a hybrid strategy:
\begin{enumerate}
    \item \textbf{Data-Driven Selection:} We calculated XGBoost feature importances and extracted the top 150 features.
    \item \textbf{Domain Feature Merging:} We merged these with the 18 bank-recommended features (Table 1) suggested by the bank's security team.
    \item \textbf{Resulting Dimension:} Compressed the input space to a 168-dimensional hybrid feature set, maintaining 99% of the original model's predictive power while reducing inference latency by 95%.
\end{enumerate}

\section{Machine Learning Model Pipeline}

Rather than using a single model, we deploy a **Dual-Engine Architecture** to balance conflicting operational objectives:

\begin{grammarbox}
\textbf{Precision Focus (XGBoost):}\ \texttt{Objective: Minimize False Alerts} $\Rightarrow$ Reached $100\%$ precision on clean profiles during validation, ensuring no legitimate customers are blocked during low-alert operational periods.\\
\textbf{Recall Focus (LightGBM):}\ \texttt{Objective: Maximize Detection} $\Rightarrow$ To mitigate the $0.89\%$ class imbalance, we tuned the booster with \texttt{max\_depth=4}, \texttt{num\_leaves=15}, and \texttt{scale\_pos\_weight=110} (penalizing misses on positive cases by 110x). Reached $93.75\%$ recall, capturing 15 out of 16 active mules.
\end{grammarbox}

\section{Interactive Sandbox \& Surrogate Scorecard}

\subsection{Profile Cloning Mechanism}
Gradient boosted tree models require fully populated feature vectors to run inference. For sandbox simulations, rather than forcing investigators to manually enter 168 columns, we engineered a \textbf{Profile Cloning} mechanism. The investigator clones a real customer profile (which populates all baseline columns) and modifies only the target 20 parameters. The backend merges the edits and passes the full vector to the model.

\subsection{Perturbation-Based Scorecard Engine}
To simulate compliance policy interventions, the backend merges model probability with a rule-based scorecard. The scorecard checks for manual perturbations (differences between sandbox values and cloned baseline values) and applies risk overrides:

\begin{mdframed}[backgroundcolor=red!3, linecolor=red!30, linewidth=0.8pt]
\textbf{Policy Escalations (Surcharges):}
\begin{itemize}[itemsep=2pt, leftmargin=15pt]
    \item Active Fraud Link flag ($F670$): adds $+35\%$ risk.
    \item High-Volume Outgoing Transfers ($F1692 \ge 6.0$): adds $+25\%$ risk.
    \item Younger Age Demographic ($F3894 \le 23$): adds $+10\%$ risk.
\end{itemize}
\end{mdframed}

\begin{mdframed}[backgroundcolor=green!3, linecolor=green!30, linewidth=0.8pt]
\textbf{Policy Mitigations (Offsets):}
\begin{itemize}[itemsep=2pt, leftmargin=15pt]
    \item Verified Occupation as Salaried ($F3891$): subtracts $-25\%$ risk.
\end{itemize}
\end{mdframed}

If a profile is cloned with zero changes, the scorecard returns $0.0$ deviation, matching the production baseline score exactly, preserving audit trail integrity.

\section{Entity Resolution Graph Linkage}

To trace money laundering syndicates, MuleTrace runs real-time entity resolution connecting accounts sharing both \textbf{Branch Code ($F3887$)} and \textbf{Registered Occupation ($F3891$)}. Nodes are visual coded by threat levels:

\begin{itemize}[itemsep=2pt]
    \item \textbf{Purple Diamond ($\blacklozenge$):} Sandbox simulations.
    \item \textbf{Red Dot ($\bullet$):} Flagged accounts ($P \ge Threshold$).
    \item \textbf{Orange Dot ($\bullet$):} Warning accounts ($20\% \le P < Threshold$).
    \item \textbf{Green Dot ($\bullet$):} Safe accounts ($P < 20\%$).
\end{itemize}

Selecting any node triggers **Focus Mode**, which dynamically fades out unrelated nodes (opacity to 15%) and highlights the suspect's immediate neighborhood ring.

\subsection{Focus Mode Opacity Calculations}
When an account node $N_{select}$ is clicked, the vis.js network engine recalculates node and edge opactity values dynamically:

\begin{equation}
\text{Node Opacity}(N) = 
\begin{cases} 
1.0 & \text{if } N = N_{select} \lor N \in \text{Neighbors}(N_{select}) \\
0.15 & \text{otherwise}
\end{cases}
\end{equation}

\begin{equation}
\text{Edge Opacity}(E) = 
\begin{cases} 
1.0 & \text{if } E \text{ is connected to } N_{select} \\
0.05 & \text{otherwise}
\end{cases}
\end{equation}

This isolates the network visual footprint to the immediate ring under investigation.

\section{Explainable AI \& Dynamic NLG}

To comply with model safety standards, we compute Local Feature Attribution (LFA) relative to clean baseline medians:

\begin{equation}
\text{Attribution}_i = (X_i - \mu_i) \times \text{Importance}_i
\end{equation}

Where $X_i$ is the input feature value, $\mu_i$ is the median feature value of the clean customer cohort, and $\text{Importance}_i$ is the global importance weight assigned by the model engine.

The \textbf{NLP Explanation Engine} maps technical feature IDs (e.g. $F670$, $F3894$) to human-readable terms and generates:
\begin{enumerate}
    \item \textbf{AI Forensic Narrative Summary}: A paragraph-long compliance synthesis summarizing the primary threat vectors.
    \item \textbf{Compliance Sentences}: Dynamic, rule-based explanations under each feature bar explaining *why* the value is anomalous (e.g., *\"Customer age (19) is below the standard low-risk threshold\"*).
\end{enumerate}

Compliance investigators can auto-generate a pre-filled, legally structured \textbf{Suspicious Activity Report (SAR)} template, reducing manual report writing time by \textbf{$90\%$}.
