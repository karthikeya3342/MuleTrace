\section*{Project Overview}
\addcontentsline{toc}{section}{Project Overview}

\textbf{MuleTrace (MTX)} is an explainable machine learning and network analysis system designed to identify money-mule accounts in real time. Mule accounts, which are used by criminal syndicates to launder illicit funds through rapid electronic transactions, represent a severe threat to commercial banking networks. MTX addresses this threat by integrating bimodal ML classification, policy auditing, and interactive network graphing.

MTX implements a complete \textbf{Two-Stage Real-Time Ingestion and Auditing} pipeline: incoming transactions are verified against dynamic baseline medians, evaluated via XGBoost/LightGBM model ensembles, subjected to compliance-based perturbation auditing, and visualized as interactive node networks.

\subsection*{Project Objectives}
\begin{itemize}[itemsep=3pt]
    \item Implement a robust ML classification pipeline addressing severe target class imbalance ($0.89\%$ positive mule rate).
    \item Audit raw transactional datasets to identify and neutralize temporal and index-based data leakages.
    \item Compress high-dimensional datasets containing 3,921 features into an optimized 168-dimensional hybrid feature set to meet sub-2ms ingestion latency SLAs.
    \item Build a bimodal decision engine comprising precision-focused and recall-focused gradient boosted tree classifiers.
    \item Engineer a compliance sandbox using profile cloning to evaluate policy adjustments without database contamination.
    \item Deploy a force-directed entity resolution graph linkage map using vis.js to isolate mule syndicate rings.
\end{itemize}

\subsection*{Technology Stack}
\begin{itemize}[itemsep=3pt]
    \item \textbf{Backend Ingest \& API:} FastAPI, Uvicorn (asynchronous event loop)
    \item \textbf{Machine Learning Engines:} XGBoost (extreme gradient boosting), LightGBM (light gradient boosting)
    \item \textbf{Data Auditing:} Pandas, NumPy, Scikit-learn
    \item \textbf{Graphics \& Web UI:} HTML5, CSS Grid/Flexbox, vis.js (force-directed canvas visualization)
    \item \textbf{Platform:} Render Cloud Platform with dynamic port binding
\end{itemize}

\subsection*{System Processing Phases}
\begin{enumerate}[itemsep=2pt]
    \item Real-Time Streaming Data Ingestion \& Median Ingestion \hfill \texttt{app.py}
    \item Preprocessing \& Categorical Encoding Mapping \hfill \texttt{app.py}, \texttt{model\_metadata.json}
    \item Dual-Engine Model Inference (XGBoost vs. LightGBM) \hfill \texttt{xgb\_hybrid\_model.json}, \texttt{lgb\_hybrid\_model.txt}
    \item Sandbox State Check \& Perturbation Scorecard Application \hfill \texttt{app.py}
    \item Local Feature Attribution (LFA) \& Explainable AI Weighting \hfill \texttt{app.py}
    \item Natural Language Compliance Narrative Generation (NLG) \hfill \texttt{app.py}
    \item Dynamic Graph Linkage Update \& Automated SAR Template \hfill \texttt{static/index.html}
\end{enumerate}

\section{Data Preprocessing \& Data Leakage Resolution}

\subsection{Leakage Auditing Rules}
\begin{grammarbox}
\textbf{Index/Sorting Rule:}\ Remove row index column. Shuffle training records using random seed. Reset index pointers to prevent gradient boosting from splitting on chronological indices.\\[4pt]
\textbf{Temporal Batch Rule:}\ Identify columns with high target correlation but zero causal connection (e.g. $F2230$, $F3912$). Exclude these temporal batch features entirely from the training pipeline.\\[4pt]
\textbf{Feature Dimension Rule:}\ Compress 3,921 anonymized features into a 168-feature hybrid set. Maintain 99\% predictive performance while meeting sub-2ms latency SLAs.
\end{grammarbox}

\subsection{Phase Description}
During initial model audits, we identified two severe data leakages that, if left untreated, would cause models to overfit and fail in production. First, the training dataset was sorted by the target variable (mule status). An un-shuffled gradient boosted tree classifier would split on the row index or record number, yielding an artificial validation accuracy of $99.9\%$ that drops to $50\%$ in production. We dropped the index, shuffled the dataset using a random seed, and reset the index pointers before model training. Second, features $F2230$ and $F3912$ acted as exact temporal separators (clean cases were clustered in specific months, while mule cases were clustered in others). We excluded $F2230$ and $F3912$ entirely from the training pipeline.

Running real-time inference on 3,921 features is computationally unfeasible under strict banking SLA limits (<2ms). We compressed the dimensions using a hybrid strategy. First, we calculated XGBoost feature importances and extracted the top 150 features. Second, we merged these with the 18 bank-recommended features suggested by fraud team experts. This resulted in a 168-dimensional hybrid feature set, maintaining $99\%$ of the original model's predictive power while reducing inference latency by $95\%$.

\section{Machine Learning Model Pipeline}

\subsection{Model Engine Architecture Rules}
\begin{grammarbox}
\begin{tabbing}
\hspace{2.8cm}\=\kill
$\langle engine \rangle$ \> $\to$ \texttt{xgboost} $\mid$ \texttt{lightgbm}\\[2pt]
$\langle objective\_xgb \rangle$ \> $\to$ \texttt{binary:logistic} $\mid$ \texttt{eval\_metric = logloss}\\[2pt]
$\langle scale\_pos\_xgb \rangle$ \> $\to$ $1.0$ (Precision-focused baseline)\\[2pt]
$\langle objective\_lgb \rangle$ \> $\to$ \texttt{binary} $\mid$ \texttt{metric = binary\_logloss}\\[2pt]
$\langle scale\_pos\_lgb \rangle$ \> $\to$ $110.0$ (Recall-focused penalty override)\\[2pt]
$\langle hyper\_lgb \rangle$ \> $\to$ \texttt{max\_depth = 4} $\mid$ \texttt{num\_leaves = 15}
\end{tabbing}
\end{grammarbox}

\subsection{Phase Description}
Rather than using a single model, we deploy a Dual-Engine Architecture to balance conflicting operational objectives:
\begin{itemize}
    \item \textbf{Precision Focus (XGBoost):} Designed to minimize false alerts. Reached $100\%$ precision on clean profiles during validation, ensuring no legitimate customers are blocked during low-alert operational periods.
    \item \textbf{Recall Focus (LightGBM):} Designed to maximize detection. To mitigate the $0.89\%$ class imbalance, we tuned the booster with \texttt{max\_depth=4}, \texttt{num\_leaves=15}, and \texttt{scale\_pos\_weight=110} (penalizing misses on positive cases by 110x). Reached $93.75\%$ recall, capturing 15 out of 16 active mules.
\end{itemize}

Both models are serialized and stored directly in the workspace directory. XGBoost is saved as a JSON model representation (\texttt{xgb\_hybrid\_model.json}), and LightGBM is exported as a lightweight text booster file (\texttt{lgb\_hybrid\_model.txt}). The FastAPI server loads these models directly into memory on startup.

\section{Interactive Sandbox \& Profile Cloning}

\subsection{Sandbox Perturbation Rules}
\begin{grammarbox}
\textbf{Profile Cloning Rule:}\ Ingestion must clone a baseline customer profile and merge specific modifications to generate a complete 168-dimensional feature vector. Zero-modification clones must reconcile exactly with their baseline.\\[4pt]
\textbf{Policy Surcharge Rule:}\ Active Fraud Link ($F670 = 1$) adds $+35\%$ to risk. High-volume transfers ($F1692 \ge 6$) add $+25\%$ to risk. Young age ($F3894 \le 23$) adds $+10\%$ to risk.\\[4pt]
\textbf{Policy Mitigation Rule:}\ Verifying occupation as salaried ($F3891$) subtracts $-25\%$ from risk.
\end{grammarbox}

\subsection{Phase Description}
Gradient boosted tree models require fully populated feature vectors to run inference. For sandbox simulations, rather than forcing investigators to manually enter 168 columns, we engineered a Profile Cloning mechanism. The investigator clones a real customer profile (which populates all baseline columns) and modifies only the target 20 parameters. The backend merges the edits and passes the full vector to the model.

To simulate compliance policy interventions, the backend merges model probability with a rule-based scorecard. The scorecard checks for manual perturbations (differences between sandbox values and cloned baseline values) and applies risk overrides. Policy escalations (surcharges) apply if a high-risk variable is set, while mitigations (offsets) apply if low-risk indicators are met. If a profile is cloned with zero changes, the scorecard returns $0.0$ deviation, matching the production baseline score exactly, preserving audit trail integrity.

\section{Entity Resolution Graph Linkage}

\subsection{Graph Linkage Resolution Rules}
\begin{grammarbox}
\textbf{Node Classification:}\\
$\bullet$ \texttt{Purple Diamond ($\blacklozenge$)} --- Sandbox Simulation Profile\\
$\bullet$ \texttt{Red Circle ($\bullet$)} --- Flagged Suspect Mule ($P \ge \text{Threshold}$)\\
$\bullet$ \texttt{Orange Circle ($\bullet$)} --- High Risk Warning ($20\% \le P < \text{Threshold}$)\\
$\bullet$ \texttt{Green Circle ($\bullet$)} --- Low Risk Safe ($P < 20\%$)\\[4pt]
\textbf{Linkage Definition:}\\
An edge exists between node $A$ and node $B$ if and only if they share the same Bank Branch ($F3887$) and Registered Occupation ($F3891$).
\end{grammarbox}

\subsection{Phase Description}
To trace money laundering syndicates, MuleTrace runs real-time entity resolution connecting accounts sharing both Branch Code ($F3887$) and Registered Occupation ($F3891$). Node coordinates are visual coded by threat levels. Selecting any node triggers Focus Mode, which dynamically fades out unrelated nodes and highlights the suspect's immediate neighborhood ring.

When an account node $N_{select}$ is clicked, the vis.js network engine recalculates node and edge opacity values dynamically:
\begin{equation}
\text{Node Opacity}(N) = 
\begin{cases} 
1.0 & \text{if } N = N_{select} \lor N \in \text{Neighbors}(N_{select}) \\
0.15 & \text{otherwise}
\end{cases}
\end{equation}

\begin{equation}
\text{Edge Opacity}(E) = 
\begin{cases} 
1.0 & \text{if } E \text{ is connected to } N_{select} \\
0.05 & \text{otherwise}
\end{cases}
\end{equation}

This isolates the network visual footprint to the immediate ring under investigation, enabling fraud analysts to track mule networks.

\section{Explainable AI \& Dynamic NLG}

\subsection{Local Feature Attribution Rules}
\begin{grammarbox}
\textbf{Attribution Formula:}\\
$\text{Attribution}_i = (X_i - \mu_i) \times \text{Importance}_i$\\[4pt]
Where $X_i$ is the sandbox value of feature $i$, $\mu_i$ is the median value of the clean customer cohort for feature $i$, and $\text{Importance}_i$ is the global model importance of feature $i$.\\[4pt]
\textbf{Dynamic NLG Narrative Rule:}\\
Format technical codes (e.g. $F670$) to human-readable labels (e.g. "Fraud Link Flag") and generate legally sound forensic summaries for the SAR filing.
\end{grammarbox}

\subsection{Phase Description}
To comply with model safety standards, we compute Local Feature Attribution (LFA) relative to clean baseline medians. The NLP Explanation Engine maps technical feature IDs (e.g. $F670$, $F3894$) to human-readable terms and generates:
\begin{enumerate}
    \item \textbf{AI Forensic Narrative Summary:} A paragraph-long compliance synthesis summarizing the primary threat vectors.
    \item \textbf{Compliance Sentences:} Dynamic, rule-based explanations under each feature bar explaining why the value is anomalous (e.g., "Customer age is below the standard low-risk threshold").
\end{enumerate}

Compliance investigators can auto-generate a pre-filled, legally structured Suspicious Activity Report (SAR) template, reducing manual report writing time by $90\%$.

\section{REST API Endpoint Specifications}

\subsection{REST API Schema Rules}
\begin{grammarbox}
\textbf{POST /predict:}\\
Input: \texttt{\{ features: \{...\}, base\_features: \{...\}, engine: "xgboost", threshold: 0.50 \}}\\[4pt]
Output: \texttt{\{ account\_id: "...", engine: "...", probability: 0.6501, prediction: 1, flagged: true, top\_drivers: [...] \}}\\[4pt]
\textbf{GET /status:}\\
Output: \texttt{\{ status: "healthy", database\_size: 10000, features\_loaded: 168, models\_loaded: ["xgboost", "lightgbm"] \}}
\end{grammarbox}

\subsection{Phase Description}
The API endpoints are served asynchronously via Uvicorn. The backend processes incoming REST requests, reconciles missing columns against baseline medians, passes the full vector to the selected ML model (XGBoost or LightGBM), evaluates the scorecard, calculates local feature attributions, and returns a compiled response. The server is configured to handle sub-2ms inferences, aligning with banking transactional speeds.
