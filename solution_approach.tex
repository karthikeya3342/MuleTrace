\section{Introduction}
Money laundering conducted via ``money mule'' accounts has emerged as one of the most critical security and compliance challenges for commercial banking institutions. Fraud syndicates recruit vulnerable individuals---such as students, housewives, or financially distressed citizens---to lease their personal bank accounts. Once illicit funds are transferred into these mule accounts, they are immediately split and transferred out via microtransactions, digital wallets, or international wires, obfuscating the audit trail.

Traditional Anti-Money Laundering (AML) systems utilize static, rule-based heuristics that trigger massive volumes of false-positive alerts, resulting in severe investigator fatigue and elevated operational overhead. Conversely, although high-capacity machine learning models (e.g., neural networks or deep gradient boosted trees) capture complex non-linear behaviors, they function as ``black boxes.'' This creates a compliance gap under regulatory frameworks (such as the US Federal Reserve Board's SR 11-7 Model Risk Management guidelines), which mandate clear, explainable decision-making logic for any automated block placed on a customer's account.

\textbf{MuleTrace (MTX)} is designed as a unified, real-time explainable machine learning decision engine. It achieves two key banking requirements:
\begin{enumerate}
    \item \textbf{High-Performance Classification:} Exposing a bimodal machine learning pipeline (XGBoost and LightGBM) to balance high precision and high recall.
    \item \textbf{Human-in-the-Loop Explainability (XAI):} Translating mathematical local feature attributions into natural language compliance narratives and offering a risk sandbox for auditing policy interventions.
\end{enumerate}

1. \textbf{Transaction Ingestion Layer:} Ingests streaming account transaction variables and queries cached historical medians to compile a complete customer profile.
2. \textbf{Bimodal Machine Learning Pipeline:} Runs parallel inference on the profile. The compliance officer can toggle between XGBoost (precision focus for lower alert volumes) and LightGBM (recall focus during threat surges).
3. \textbf{Perturbation-Based Scorecard Engine:} Checks if the account is in a sandbox state. If modifications are detected, it compares parameters against baseline records and applies risk overrides and mitigating credits.
4. \textbf{XAI \& NLP Explanation Engine:} Calculates local feature contributions and dynamically generates human-readable forensic arguments.
5. \textbf{Web Dashboard:} Displays risk profiles, auto-populates Suspicious Activity Report (SAR) templates, and plots the vis.js network linkage map.

\section{Dataset Architecture \& Column Mappings}

\subsection{Raw Data Structure}
The raw training dataset contains high-dimensional, sparse behavioral histories comprising \textbf{3,921 raw features} with an extremely low target class distribution (money mules represent only \textbf{0.89\%} of the total database).

\subsection{Column Naming Strategy}
To protect customer privacy and prevent the exposure of proprietary transaction routing variables, the raw dataset columns are anonymized using sequential index codes (e.g., $F1, F2, \dots, F3921$). 

To construct an explainable compliance engine, we performed a statistical correlation audit to isolate the \textbf{18 bank-recommended features} suggested by fraud team experts. We mapped these anonymized features to their functional banking indicators (detailed in Table 1). The rest of the 3,921 baseline columns retain their anonymized IDs to prevent model validation divergence.

% ==========================================
\section{Preprocessing \& Data Leakage Resolution}
During initial model audits, we identified two severe data leakages that, if left untreated, would cause models to overfit and fail in production.

\subsection{Index/Sorting Leakage}
The training dataset was sorted by the target variable (mule status). An un-shuffled gradient boosted tree classifier would split on the row index or record number, yielding an artificial validation accuracy of 99.9\% that drops to 50\% in production.
\\\textbf{Mitigation:} We dropped the index, shuffled the dataset using a random seed, and reset the index pointers before model training.

\subsection{Temporal Batch Leakage}
Features $F2230$ and $F3912$ acted as exact temporal separators (clean cases were clustered in specific months, while mule cases were clustered in others). The model learned to classify based on date batches rather than transactional behaviors.
\\\textbf{Mitigation:} We excluded $F2230$ and $F3912$ entirely from the training pipeline.

\section{Machine Learning Model Pipeline}
Rather than using a single model, we deploy a **Dual-Engine Architecture** to balance conflicting operational objectives:

1. \textbf{XGBoost (Precision Focus):} Designed for periods of high compliance backlog. Optimizes for a low false-positive rate. Achieved \textbf{100\% Precision} during validation on clean profiles, ensuring no legitimate customers are blocked.
2. \textbf{LightGBM (Recall Focus):} Designed for active threat surges. To mitigate the 0.89\% class imbalance, we tuned the booster with `max_depth=4`, `num_leaves=15`, and `scale_pos_weight=110` (penalizing misses on positive cases by 110x). Achieved \textbf{93.75\% Recall}, capturing 15 out of 16 active mules.

\section{Interactive Sandbox \& Graph Entity Resolution}

\subsection{Profile Cloning Mechanism}
Ensemble trees require fully populated feature vectors to run inference. For sandbox simulations, we engineered a \textbf{Profile Cloning} mechanism. The investigator clones a real customer profile (which populates all baseline columns) and modifies only the target 20 parameters. The backend merges the edits and passes the full vector to the model.

\subsection{Perturbation-Based Scorecard Engine}
To simulate compliance policy interventions, the backend merges model probability with a rule-based scorecard:
\begin{itemize}
    \item \textbf{Policy Escalations (Surcharges):} Active Fraud Link flags ($F670$) add $+35\%$; high-volume transfers ($F1692 \ge 6.0$) add $+25\%$; younger age profiles ($F3894 \le 23$) add $+10\%$ risk.
    \item \textbf{Policy Mitigations (Offsets):} Verifying occupation as salaried ($F3891$) subtracts $-25\%$.
    \item \textbf{Reconciliation Integrity:} If a profile is cloned with zero changes, the scorecard returns $0.0$ deviation, matching the production baseline score exactly.
\end{itemize}

\subsection{Entity Resolution Graph Linkage}
To trace syndicates, MTX runs real-time entity resolution connecting accounts sharing both \textbf{Branch Code ($F3887$)} and \textbf{Registered Occupation ($F3891$)}. 
Nodes are visual coded by threat levels:
\begin{itemize}
    \item \textbf{Purple Diamond ($\blacklozenge$):} Sandbox simulations.
    \item \textbf{Red Dot ($\bullet$):} Flagged accounts ($P \ge Threshold$).
    \item \textbf{Orange Dot ($\bullet$):} Warning accounts ($20\% \le P < Threshold$).
    \item \textbf{Green Dot ($\bullet$):} Safe accounts ($P < 20\%$).
\end{itemize}
Selecting any node triggers Focus Mode, which dynamically fades out unrelated nodes (opacity to $15\%$) and highlights the suspect's immediate neighborhood ring.

\section{Explainable AI \& Dynamic NLG}
To comply with model safety standards, we compute Local Feature Attribution (LFA) relative to clean baseline medians:
\begin{equation}
\text{Attribution}_i = (X_i - \mu_i) \times \text{Importance}_i
\end{equation}

The \textbf{NLP Explanation Engine} maps technical feature IDs (e.g. $F670$, $F3894$) to human-readable terms and generates:
1. \textbf{AI Forensic Narrative Summary}: A paragraph-long compliance synthesis summarizing the primary threat vectors.
2. \textbf{Compliance Sentences}: Dynamic, rule-based explanations under each feature bar explaining *why* the value is anomalous (e.g., *\"Customer age (19) is below the standard low-risk threshold\"*).

Compliance investigators can auto-generate a pre-filled, legally structured \textbf{Suspicious Activity Report (SAR)} template, reducing manual report writing time by 90\%.
