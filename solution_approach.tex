\\begin{abstract}
Money laundering conducted through third-party \"money mule\" accounts represents a systemic threat to global banking operations. Traditional static rule-based engines suffer from elevated false-positive rates and fail to capture sophisticated, evolving behavioral patterns. Conversely, deep black-box neural networks are legally restricted under banking model risk management frameworks due to their lack of explainability. This paper presents \\textbf{MuleTrace (MTX)}, an end-to-end explainable artificial intelligence (XAI) classification platform. We neutralize critical data leakages (sorting and temporal batch artifacts) in a highly dimensional dataset (3,921 features) and implement a 168-dimensional hybrid feature compression. We introduce a dual-model decision engine comprising a precision-focused XGBoost classifier ($100\\%$ precision) and a recall-focused LightGBM classifier ($93.75\\%$ recall). Finally, we present an interactive sandbox utilizing a perturbation-based surrogate scorecard, an NLP-driven Explanation Engine, and a force-directed entity resolution graph linkage map to trace mule syndicates in real-time.
\\end{abstract}

\\begin{IEEEkeywords}
Explainable AI, Anti-Money Laundering, Feature Selection, Data Leakage, Gradient Boosting, Entity Resolution, Force-Directed Graph.
\\end{IEEEkeywords}

\\section{Introduction}
\\IEEEcommandreftext{F}inancial institutions face strict regulatory requirements to monitor and report suspicious account behaviors. Money mules, individuals who lease or sell their bank accounts to criminal syndicates to transfer laundered funds, are highly difficult to detect because their transactions often mimic standard retail customer activity. 

This solution approach describes the design, math, and architecture of \\textbf{MuleTrace (MTX)}, a real-time AML platform that bridges high-performance machine learning classification with regulatory explainability and interactive compliance sandboxing.

\\section{Dataset Characterization \\& Preprocessing Methodology}

\\subsection{Raw Dataset Properties}
The raw dataset consists of transaction-level behavioral logs with the following characteristics:
\\begin{itemize}
    \\item \\textbf{Dimensionality:} 3,921 columns representing raw customer transactional and profile attributes.
    \\item \\textbf{Class Imbalance:} Extremely skewed; positive instances (money-mule accounts) represent only \\textbf{$0.89\\%$} of the dataset, demanding robust class-weighting and regularization to prevent majority-class bias.
\\end{itemize}

\\subsection{Critical Data Leakage Neutralization}
A model audit revealed two severe data leakages in the raw dataset that, if left untreated, would cause models to overfit and fail in a production environment:
\\begin{enumerate}
    \\item \\textbf{Index/Sorting Leakage:} The raw CSV dataset rows were sorted chronologically by the target classification variable. Standard models would learn row indices as a shortcut, leading to artificial validation accuracy.
    \\\\\\textit{Mitigation:} We dropped the index column, shuffled the dataset using a random seed, and reset the index pointers before model training.
    \\item \\textbf{Temporal Batch Leakage:} Feature $F2230$ and feature $F3912$ acted as exact temporal separators (clean cases clustered in specific batches, while mule cases clustered in others). 
    \\\\\\textit{Mitigation:} We completely excluded features $F2230$ and $F3912$ from the training set, forcing the classifier to learn genuine transactional markers.
\\end{enumerate}

\\subsection{Hybrid Feature Engineering}
To meet the strict real-time banking Service Level Agreement (SLA) of sub-2ms latency, we compressed the 3,921-dimensional input space into a **168-dimensional hybrid feature set**:
\\begin{enumerate}
    \\item \\textbf{Data-Driven Selection:} Ran a baseline XGBoost model on the leakage-free dataset to extract the top 150 features ranked by gain-based feature importance.
    \\item \\textbf{Domain Knowledge Intersect:} Selected the **18 bank-recommended features** suggested by fraud detection experts and merged them with the 150 data-driven features.
\\end{enumerate}

\\section{Model Architecture \\& Dual-Engine Strategy}

We deploy a **Dual-Engine Architecture** rather than forcing a single model to handle conflicting operational objectives:

\\subsection{XGBoost Classifier (Precision Focus)}
Tuned for low-alert operational periods to eliminate false positives and reduce manual compliance backlogs.
\\begin{itemize}
    \\item \\textbf{Performance:} Reached \\textbf{$100\\%$} precision on clean profiles during validation, ensuring zero false alarms for legitimate customers.
\\end{itemize}

\\subsection{LightGBM Classifier (Recall Focus)}
Tuned for high-threat operational periods to maximize detection rates of active money mules. We mitigated the class imbalance by configuring:
\\begin{itemize}
    \\item `max_depth = 4`, `num_leaves = 15`, `min_child_samples = 8` to restrict overfitting.
    \\item `scale_pos_weight = 110` to increase the penalty of missing a positive (mule) case by 110x.
    \\item \\textbf{Performance:} Reached \\textbf{$93.75\\%$} recall (intercepting 15 out of 16 money-mule accounts).
\\end{itemize}

\\section{Compliance Sandbox \\& Graph-Based Entity Resolution}

\\subsection{Profile Cloning Mechanism}
Gradient boosted tree models require fully populated feature vectors to run inference. For sandbox simulations, rather than forcing investigators to manually enter 168 columns, we engineered a \\textbf{Profile Cloning} mechanism. The investigator clones a real customer profile (which populates all baseline columns) and modifies only the target 20 parameters. The backend merges the edits and passes the full vector to the model.

\\subsection{Perturbation-Based Scorecard Engine}
To simulate compliance policy interventions, the backend merges model probability with a rule-based scorecard:
\\begin{itemize}
    \\item \\textbf{Policy Escalations (Surcharges):} Active Fraud Link flags ($F670$) add $+35\\%$; high-volume transfers ($F1692 \\ge 6.0$) add $+25\\%$; younger age profiles ($F3894 \\le 23$) add $+10\\%$ risk.
    \\item \\textbf{Policy Mitigations (Offsets):} Verifying occupation as salaried ($F3891$) subtracts $-25\\%$.
    \\item \\textbf{Reconciliation Integrity:} If a profile is cloned with zero changes, the scorecard returns $0.0$ deviation, matching the production baseline score exactly.
\\end{itemize}

\\subsection{Entity Resolution Graph Linkage}
To trace syndicates, MTX runs real-time entity resolution connecting accounts sharing both \\textbf{Branch Code ($F3887$)} and \\textbf{Registered Occupation ($F3891$)}. 
Nodes are visual coded by threat levels:
\\begin{itemize}
    \\item \\textbf{Purple Diamond ($\\blacklozenge$):} Sandbox simulations.
    \\item \\textbf{Red Dot ($\\bullet$):} Flagged accounts ($P \\ge Threshold$).
    \\item \\textbf{Orange Dot ($\\bullet$):} Warning accounts ($20\\% \\le P < Threshold$).
    \\item \\textbf{Green Dot ($\\bullet$):} Safe accounts ($P < 20\\%$).
\\end{itemize}
Selecting any node triggers **Focus Mode**, which dynamically fades out unrelated nodes (opacity to $15\\%$) and highlights the suspect's immediate neighborhood ring.

\\section{Explainable AI (XAI) \\& NLP Summaries}
To comply with model safety standards, we compute Local Feature Attribution (LFA) relative to clean baseline medians:
\\begin{equation}
\\text{Attribution}_i = (X_i - \\mu_i) \\times \\text{Importance}_i
\\end{equation}

The \\textbf{NLP Explanation Engine} maps technical feature IDs (e.g. $F670$, $F3894$) to human-readable terms and generates:
1. \\textbf{AI Forensic Narrative Summary}: A paragraph-long compliance synthesis summarizing the primary threat vectors.
2. \\textbf{Compliance Sentences}: Dynamic, rule-based explanations under each feature bar explaining *why* the value is anomalous (e.g., *\"Customer age (19) is below the standard low-risk threshold\"*).

Compliance investigators can auto-generate a pre-filled, legally structured \\textbf{Suspicious Activity Report (SAR)} template, reducing manual report writing time by \\textbf{$90\\%$}.
